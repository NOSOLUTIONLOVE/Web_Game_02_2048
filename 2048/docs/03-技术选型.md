## 一、技术选型原则

本项目在 snake 模板基础上扩展，遵循以下原则：

1. **三层架构严格分离**：engine（TS 类）/ state（Zustand）/ UI（React）
2. **可复用**：技术选型复用 snake 的栈，最大限度降低学习成本
3. **DOM 优先**：2048 网格规模小，DOM + CSS Grid 性能足够且动画好做
4. **零后端**：纯静态前端 + Zustand persist
5. **学习价值**：覆盖 TypeScript 严格模式、Framer Motion、CSS Grid 布局等中级技能

## 二、核心选型决策

### 2.1 语言：TypeScript ✅

继承 snake 决策，5.x 严格模式（`noImplicitAny` / `strictNullChecks` / `noUnusedLocals`）。

---

### 2.2 构建工具：Vite ✅

继承 snake 决策，5.x 版本，配置 `base: './'` 以兼容静态部署。

---

### 2.3 渲染方案：DOM + CSS Grid ✅

| 方案 | 适用场景 | 评分 |
| --- | --- | :---: |
| **DOM + CSS Grid** ✅ | 4×4 等小规模网格 + 动画 | ⭐⭐⭐⭐⭐ |
| Canvas 2D | 大规模粒子 / 像素游戏 | ⭐⭐⭐ |
| WebGL | 复杂 2D / 3D | ⭐⭐ |

**结论**：✅ 选择 DOM + CSS Grid
**理由**：
- 4×4 仅有 16 个 DOM 节点，性能完全够
- Framer Motion 的 `layout` prop 配合 `position: absolute` 实现移动动画
- 不需要主循环，2048 是输入驱动游戏

**对比 snake**：snake 用 Canvas 2D（蛇身需要逐节绘制），2048 改 DOM（每个 tile 是独立元素，动画更简单）。

---

### 2.4 状态管理：Zustand ✅

| 方案 | 适用 | 结论 |
| --- | --- | --- |
| Redux / Zustand | 多组件共享状态 | ✅ Zustand 简洁 |
| 单文件 Board 类 | 仅 1 个组件订阅 | ❌ 跨组件订阅不便 |
| Context + useReducer | 简单共享 | 🟡 性能差 |

**结论**：✅ 选择 Zustand（继承 snake 决策）

**差异**：snake 用 Zustand 仅持久化 highScore，2048 还需持久化 audioEnabled，并暴露 `applyBoard` 回调供 GameEngine 调用。

---

### 2.5 动画：Framer Motion ✅

| 方案 | 适用 | 结论 |
| --- | --- | --- |
| **Framer Motion** ✅ | 复杂多态动画 | ✅ `layout` 自动算位置 |
| CSS Transition | 简单状态切换 | 🟡 手写 keyframes |
| Web Animations API | 精细控制 | 🔴 API 复杂 |

**结论**：✅ 选择 Framer Motion（继承 snake 决策）

**核心机制**：
```tsx
<motion.div
  layout
  animate={{ top, left, scale: isMerged ? [1, 1.2, 1] : 1 }}
  transition={{ type: 'spring', stiffness: 500, damping: 35 }}
/>
```

---

### 2.6 UI 组件：shadcn/ui ✅

继承 snake 决策：shadcn/ui + Tailwind CSS + Radix UI 原语 + Lucide Icons。

**额外组件**：本项目新增的 modal 类用 Radix Dialog 包装，settings 用 Radix Switch。

---

### 2.7 样式：Tailwind CSS ✅

继承 snake 决策：暗黑紫调色板（zinc-950 背景 + primary 紫色 + accent 青色）。

**特点**：
- 方块区（Board）保持经典 2048 暖色（`#bbada0` 背景 + 各 value 独立色）
- 外层 UI（HUD / 菜单 / 弹窗）用紫调，与项目设计语言一致

---

### 2.8 输入：键盘 + 触屏 ✅

继承 snake 决策：单一 `Input` 类绑定 keydown / touchstart / touchend / touchmove。

**差异**：snake 触屏绑在 canvas DOM 节点，2048 绑在 Board 容器 div（`useEffect` 中 `engine.getInput().bind(cbs, boardRef.current)`）。

---

### 2.9 音效：Web Audio API ✅

继承 snake 决策：Oscillator + Gain 合成，6 种事件音效（move/merge/spawn/win/lose/click）。

---

### 2.10 部署：Vercel ✅

继承 snake 决策。

## 六、关键技术决策点

### 决策点 1：渲染用 DOM 还是 Canvas？

| 方案 | 优势 | 劣势 |
| --- | --- | --- |
| **DOM + CSS Grid** ✅ | 动画简单（Framer Motion `layout`） | 节点数有限 |
| Canvas 2D | 性能极限高 | 动画要手动算位置，复杂 |

**决策**：✅ DOM + CSS Grid（PRD §9 明确推荐，4×4 规模完全够用）

### 决策点 2：移动算法如何处理四个方向？

| 方案 | 优势 | 劣势 |
| --- | --- | --- |
| **旋转到 left 统一处理** ✅ | 代码量少，无重复 | 坐标转换需小心 |
| 4 个独立函数 | 直观 | 重复代码 4 倍 |

**决策**：✅ 旋转到 left（Board.ts 内部用 `rotateClockwise` / `rotateCounterClockwise` / `rotateHorizontal`）

### 决策点 3：动画驱动逻辑还是逻辑驱动动画？

| 方案 | 机制 |
| --- | --- |
| **状态驱动** ✅ | Engine 返回 MoveResult（含 merged / spawned），UI 据此给 tile 加 `isMerged` / `isNew` 标记，触发额外动画 |
| 监听 DOM 变化 | 复杂，状态与动画耦合 |

**决策**：✅ 状态驱动（`useGameStore.lastMove` 携带 MoveResult）

### 决策点 4：撤销栈存什么？

| 方案 | 粒度 |
| --- | --- |
| **整盘快照** ✅ | 每次移动前深拷贝 grid + score + moves |
| 增量（操作列表） | 节省内存，逻辑复杂 |

**决策**：✅ 整盘快照（最多 10 份，4×4 = 16 cells，每份几 KB，可接受）

### 决策点 5：是否用 Web Worker？

| 方案 | 适用 |
| --- | --- |
| **主线程同步** ✅ | 4×4 移动算法 O(16)，远低于 1ms 预算 |
| Web Worker | 复杂 AI / 搜索算法 |

**决策**：✅ 不需要 Worker（贪吃蛇 / 2048 都不需要）
